# プロジェクト運用ルール

> 基本方針はトップディレクトリの `claude.md` に準拠する。
> 本ファイルはこのプロジェクト固有の補足ルールを定義する。

---

## プロジェクト概要

**開発プロジェクト管理 AIエージェント**

要件定義書・改造依頼書の読み込みから、リポジトリ判定・機能要件抽出・ブランチ設計・担当者アサイン・コードレビュー自動実行・進捗管理までを一貫して支援するWebアプリケーション。各フェーズに人間の承認ポイントを設けることで、AI任せにしすぎず現実的な運用を実現する。

| 項目 | 内容 |
|---|---|
| 目的 | DevOps×AIエージェント ハッカソン 提出 |
| フロントエンド | Next.js（TypeScript） |
| デプロイ先 | Google Cloud Run |
| 主なAPI | Gemini 2.0 Flash API / Gemini Embedding API |
| DB | Firestore（Vector Search含む） |
| 外部連携 | GitHub API / Gmail API |

---

## システムの全体フロー

```
要件定義書・改造依頼書（PDF / Word / テキスト）
  │
  ▼
① リポジトリ判定フェーズ
  │ 新規 / 改造を判定
  │
  ├─【改造】Firestoreの機能仕様データと照合
  │          → 該当リポジトリの候補を提示
  │          → 担当者が確認・確定  ↑ 人間の介入ポイント①-A
  │
  └─【新規】ベクトル類似度検索で類似リポジトリを検索
             → 類似候補を提示（なければ完全新規として表示）
             → 管理者が確認・決定  ↑ 人間の介入ポイント①-B
  │
  ▼
② 機能要件抽出フェーズ
  │ AIが要件定義書から要件の叩き台を自動抽出
  │ 担当者が修正・追加・削除して機能要件票を確定
  │   ↑ 人間の介入ポイント②
  │   ※ 担当者が手動作成した機能要件票でここから開始することも可
  ▼
③ ブランチ設計・担当割り当てフェーズ
  │ 確定した機能要件票をもとにブランチを設計
  │ 担当者をアサイン
  │   ↑ 人間の介入ポイント③（承認）
  ▼
④ リポジトリ・ブランチ自動作成
  │ GitHub API でリポジトリ・main・各作業ブランチを作成
  │ 担当者に作業依頼メールを送信
  ▼
⑤ 開発作業（担当者が各ブランチで実装）
  │
  │ コミット発生
  ▼
⑥ コードレビュー自動実行
  │ ブランチの目標仕様（機能要件票）を参照しながらレビュー
  │ レビュー結果を担当者にメール通知
  │ 進捗ステータスを更新
  ▼
⑦ 進捗ダッシュボード
    各ブランチの状態・コミット数・レビュー結果を一覧表示
```

## フェーズ別の仕様詳細

### ① リポジトリ判定フェーズ

Gemini APIで要件定義書を解析し、新規開発か改造かを判定する。

**【改造の場合】**
- Firestoreに蓄積された機能仕様データとテキスト照合
- システム名・機能名が明記されていれば高精度で特定可能
- 曖昧な場合は複数候補を提示して担当者が選択

**【新規の場合】**
- Gemini Embedding APIで要件定義書をベクトル化
- Firestoreに保存済みの既存リポジトリ概要ベクトルと類似度検索
- 類似候補をスコア順に提示し、管理者が確認・決定
- 完全に類似なしの場合もその旨を明示して管理者が判断

> Firestoreに蓄積するリポジトリデータ（概要・機能要件リスト・要件定義書の要約テキスト・ベクトル）を、進捗管理とリポジトリ検索の両方に共用する。

### ② 機能要件抽出フェーズ

AIが要件定義書を解析して機能要件の叩き台を自動生成する。ただし仕様書の記述品質によって精度が左右されるため、担当者が手動で作成した機能要件票を使って③に進むことも可能とする。

- AIの提案はあくまで叩き台として扱う
- 担当者が確定した機能要件票がドキュメントとして残るため、後から「なぜこのブランチを切ったか」の根拠を追跡できる

### ③ ブランチ設計・担当割り当てフェーズ

確定済みの機能要件票をもとに、どのブランチにどの機能を割り振るかを設計する。担当者のアサインもここで行う。

- 担当者の承認後に④へ自動移行

### ④ リポジトリ・ブランチ自動作成

- GitHub API でリポジトリ・mainブランチ・各作業ブランチを自動作成
- アサインされた担当者へ作業依頼メールを自動送信（担当ブランチ・機能要件を記載）

### ⑤⑥ コードレビュー自動実行

コミットのたびに以下を自動実行する。

- GitHub APIでdiffを取得
- そのブランチの機能要件票を参照しながらGemini APIでレビュー（目的が明確なため精度が高い）
- 仕様との乖離があれば指摘
- レビュー結果を担当者にメール通知
- Firestoreの進捗ステータスを更新

### ⑦ 進捗ダッシュボード

- 各ブランチの状態（未着手・作業中・レビュー中・完了）
- コミット数・レビュー通過回数
- 担当者別の作業状況

## Firestoreのデータ構造（概要）

進捗管理とリポジトリ検索を兼用するリポジトリ単位のドキュメント。

```
repositories/{repo_id}
  ├── repo_name          // リポジトリ名
  ├── summary            // システム概要
  ├── feature_list       // 機能要件リスト
  ├── tech_stack         // 使用技術スタック
  ├── is_new             // 新規 / 改造
  ├── requirements_text  // 要件定義書の要約テキスト
  └── embedding_vector   // ベクトル（類似度検索用）
```

## 使用コンポーネント

| コンポーネント | 役割 |
|---|---|
| GitHub Webhook | コミット検知 |
| GitHub API | リポジトリ・ブランチ作成、diff取得 |
| Cloud Run（Next.js） | WebUI・API・Webhook受信・処理のオーケストレーション |
| Gemini 2.0 Flash API | 新規/改造判定・機能要件抽出・コードレビュー実行 |
| Gemini Embedding API | 要件定義書・リポジトリ概要のベクトル化 |
| Firestore + Vector Search | 機能仕様・進捗・ベクトルの保存と類似度検索 |
| Gmail API | 作業依頼・レビュー結果のメール通知 |

## 開発方針

- **ハッカソン提出が目標**のため、完成度より動作するプロトタイプを優先する
- Haruが**コードを理解しながら自分で実装する**ことを前提とする
- Claudeは実装の提案・解説・レビューに徹し、コードの自動適用は行わない
- 不明な点・迷った点はその都度相談する

---

## コード提案のルール

- 提案コードには必ず**何をしているかのコメント**を添える
- 複数の実装方針がある場合は選択肢を提示し、採用はHaruが判断する
- TypeScript / Next.js に不慣れな部分は、関連する概念も合わせて説明する
- 一度に大量のコードを提案せず、**小さな単位で段階的に**進める

## 実装フェーズ

実装はフェーズ単位で進める。各フェーズが動作確認できてから次へ進む。

| フェーズ | 内容 |
|---|---|
| Phase 1 | プロジェクト初期設定・Cloud Run へのデプロイ確認 |
| Phase 2 | リポジトリ判定（新規/改造・ベクトル類似度検索） |
| Phase 3 | 機能要件抽出・機能要件票の編集UI |
| Phase 4 | ブランチ設計・担当者アサイン・GitHub API連携 |
| Phase 5 | コードレビュー自動実行（Webhook・Gemini API） |
| Phase 6 | 進捗ダッシュボード |
